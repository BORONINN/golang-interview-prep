## Введение

В этом материале мы рассмотрим дополнительные примитивы синхронизации в Go, которые расширяют возможности базовых мьютексов и wait groups. Будут рассмотрены: `Once` для однократного выполнения, `Cond` для условной синхронизации, `Map` для потокобезопасных операций с картами и `Pool` для эффективного управления временными объектами. Эти инструменты помогают решать сложные задачи параллелизма, оптимизировать производительность и избегать распространенных ошибок синхронизации.

Поскольку `WaitGroup`, `Mutex` и `RWMutex` уже были рассмотрены на предыдущих лекциях, перейдем к другим примитивам синхронизации.

## Once

`Once` — это механизм синхронизации, который позволяет выполнить определенную функцию только один раз. Он используется для инициализации глобальных переменных или других ресурсов, которые должны быть инициализированы только один раз.

```go
var fileData []byte  
var once sync.Once  
  
func loadFile() {  
    fileData = make([]byte, 1024)  
}  
  
func getFileData() []byte {  
    once.Do(loadFile)  
    return fileData  
}  
```
**Замечание:** Не стоит использовать `Once` для предотвращения повторных вызовов функций при наличии гонки данных. Например, если в событийной системе начали дублироваться события, `Once` может исправить ситуацию, но это будет костылем, который устраняет только симптом, а не причину.

## Cond

`Cond` — это механизм синхронизации, который позволяет заблокировать выполнение программы до наступления определенного условия (условная переменная). Обычно используется для того, чтобы горутины могли дождаться определенного условия перед продолжением выполнения.

```go
var cond = sync.Cond{L: &sync.Mutex{}}  
var data string  
var loaded bool  

func loadData() {  
    fmt.Println("Загрузка данных")  


    data = "Данные, которые очень долго ждали"  
    loaded = true  
    cond.Signal()  
}  
  
func getData() string {  
    cond.L.Lock()  
    for !loaded {  
        cond.Wait()  
    }  
    cond.L.Unlock()  
    return data  
}  
  
func main() {  
    go loadData()  
    fmt.Println(getData())  
}  
```

Когда функция `` `Wait()` `` возвращается, необходимо проверять условие в цикле, потому что пока горутина просыпалась, условие могло измениться обратно. Поэтому рекомендуется использовать `` `Wait()` `` в цикле.

Функции `` `Signal` `` и `` `Broadcast` `` используются для пробуждения спящих горутин:

- `` `Signal` `` разбудит первую горутину в очереди
- `` `Broadcast` `` разбудит все горутины

Важно сначала изменить условие, а только потом будить горутины. Механизм пробуждения реализован на структуре `notifyList`, которая использует систему уведомлений на основе тикетов.

## Map

`Map` — потокобезопасная обертка над обычной мапой. Предоставляет методы для записи, чтения, удаления, сравнения и итерирования.

```go
m := sync.Map{}  


m.Store("key", "value")  
  
v, ok := m.Load("key")  
if ok {  
    fmt.Println(v)  
}  
```
Этот тип оптимизирован для двух сценариев использования:

- Когда запись по ключу производится один раз, но читается многократно
- Когда разные горутины работают с непересекающимися ключами

Структура использует две внутренние карты: `read` (только для чтения) и `dirty` (для записи и чтения). Это позволяет оптимизировать операции чтения, которые часто не требуют блокировки.

При выполнении операции `Load` `sync.Map` сначала пытается найти значение в `read-мапе`. Если значение найдено, оно возвращается без блокировки. Если не найдено, увеличивается счетчик `misses` и проверяется `dirty-мапа` с захватом мьютекса.

Метод `` `LoadOrStore` `` позволяет избежать дополнительной блокировки при проверке наличия ключа и его сохранении. В остальных случаях комбинация обычной мапы с мьютексом или `RWMutex` может быть эффективнее.

## Pool

`Pool` — это синхронизирующий примитив для управления пулом временных объектов в Go. Он помогает сократить выделение памяти и нагрузку на сборщика мусора, позволяя использовать объекты повторно.

Самый частый пример использования — пул соединений. При выполнении запроса в сеть или к БД соединение берется из пула, используется, а затем возвращается обратно. В высоконагруженных приложениях это предотвращает постоянное пересоздание и очистку соединений.

```go
var bytePool = sync.Pool{  
    New: func() interface{} {  
        return make([]byte, 1024) // 1KB slice  
    },  
}  

func main() {  
    b := bytePool.Get().([]byte)  
    defer bytePool.Put(b)  

    for i := range b {  
        b[i] = byte(i % 256)  
    }  

    fmt.Println("Used data:", b[:10])  
}  

```
Использование пула для долгоживущих объектов может быть нецелесообразным и привести к большим затратам, чем создание отдельных объектов. Рекомендуется замерять производительность до и после внедрения пула для выбора оптимального варианта.
