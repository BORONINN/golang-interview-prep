## Введение

`` `errgroup` `` — это пакет из библиотеки `` `golang.org/x/sync` ``, который предоставляет механизм для управления группой горутин с обработкой ошибок. Он расширяет возможности стандартного `` `sync.WaitGroup` ``, добавляя централизованную обработку ошибок, ограничение параллелизма и поддержку контекста для отмены операций.

`` `errgroup` `` может быть полезен в следующих сценариях:

- Параллельная обработка данных — когда нужно обработать несколько элементов параллельно и собрать возможные ошибки
- Ограничение ресурсов — когда необходимо ограничить количество одновременно выполняемых операций
- Групповая отмена — когда ошибка в одной горутине должна привести к остановке всех остальных операций
- Упрощение кода — когда нужно избежать сложной синхронизации и обработки ошибок из `multiple` горутин

## Сравнение errgroup и sync.WaitGroup

На первый взгляд `` `errgroup` `` очень похожа на `` `sync.WaitGroup` ``. Мы так же запускаем несколько горутин и дожидаемся их завершения.

**Пример с errgroup:**

```go
var g errgroup.Group  
  
g.Go(func() error {  
    // ...  
})  
  
err := g.Wait()  

```
**Пример с sync.WaitGroup:**
```go
wg := sync.WaitGroup{}  
wg.Add(1)  

go func() {  
    defer wg.Done()  
    // ...  
}()  

wg.Wait()  

```
## Проблема обработки ошибок

Но что если нам нужно не только запустить операции, но и обработать ошибки?

Код с `` `WaitGroup` `` в лучшем случае будет выглядеть так:

```go


var goroutineErr error  
var wg sync.WaitGroup  
wg.Add(1)  
  
go func() {  
    defer wg.Done()  
      
    err := doSomething()  
    if err != nil {  
        goroutineErr = err  
    }  
}()  
  
wg.Wait()  
  
if goroutineErr != nil {  
    // обрабатываем ошибку  
}  
```

Согласитесь, не очень удобно. А если добавить еще несколько горутин?

## Решение с errgroup

Вот тут проявляется отличие `` `errgroup` `` (не просто же так она так называется):

```go


var g errgroup.Group  
  
g.Go(func() error {  
    return errors.New("first error")  
})  
  
g.Go(func() error {  
    return errors.New("second error")  
})  
  
fmt.Println(g.Wait())  
```
Это позволяет нам запускать несколько горутин со своей логикой и затем централизованно обрабатывать ошибки.

`` `g.Wait()` `` дожидается завершения всех запущенных горутин и возвращает первую ошибку (если она была).

## Дополнительные возможности errgroup

Но это еще не все преимущества `` `errgroup` ``. Она поддерживает работу с контекстом и позволяет лимитировать количество используемых горутин:

```go
g, ctx := errgroup.WithContext(context.Background())  
g.SetLimit(1)  

g.Go(func() error {  
    return DoSomething1(ctx)  
})  

g.Go(func() error {  
    return DoSomething2(ctx)  
})  

fmt.Println(g.Wait())  

```
Это очень удобно, потому что:

- Не нужно создавать собственные решения для лимитирования количества горутин
- Возможность отменить все операции через контекст — очень полезная функция

## Реализация лимитирования

Лимитация реализована через буферизированный канал (семафор):

```go
func (g *Group) Go(f func() error) {  
    if g.sem != nil {  
        g.sem <- token{}  
    }  
    // ...  
}  

```
## Когда что использовать

Если вам нужно просто запустить несколько операций параллельно и дождаться их завершения без обработки ошибок, то достаточно обычной `` `WaitGroup` ``.

`` `errgroup` `` идеально подходит для случаев, когда:

- _Нужна централизованная обработка ошибок из нескольких горутин_ с возвратом первой возникшей ошибки
- _Требуется автоматическая отмена операций_ — при возникновении ошибки в одной горутине все остальные операции автоматически отменяются через контекст. Это ключевая особенность `errgroup.WithContext()`
- _Необходимо ограничить количество одновременно выполняемых горутин_ с помощью `` `SetLimit()` ``
