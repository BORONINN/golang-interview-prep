### Интерфейсы в Go

Интерфейс — это специальный тип, который определяет набор методов, но не содержит их реализацию. Другими словами, интерфейсы описывают поведение типов, а реализация может отличаться. Это позволяет абстрагироваться от конкретных реализаций и писать более гибкий код.

### Утиная типизация

В Go используется утиная типизация (duck-typing или неявная типизация):

Если это выглядит как утка, плавает как утка и крякает как утка, то это, вероятно, и есть утка.

Определение факта реализации определенного интерфейса происходит без явного указания или наследования этого интерфейса, а просто по реализации полного набора его методов.

Другими словами, если тип реализует все методы интерфейса, значит он удовлетворяет этому интерфейсу:

```go
type Writer interface {  
    Write(data []byte)  
}  

type buffer struct {  
    data []byte  
}  

func (b *buffer) Write(data []byte) {  
    b.data = append(b.data, data...)  
}  

type logger struct{}  

func (l *logger) Write(data []byte) {  
    fmt.Printf("%s\n", string(data))  
}  

func main() {  
    b := &buffer{}  
    b.Write([]byte("hello world"))  

    l := &logger{}  
    l.Write([]byte("hello world"))  
}  

```
### Зачем нужен интерфейс

Можно спросить “а зачем нам тут интерфейс вообще?”. Например, мы можем объявить функцию, которая будет принимать интерфейс, а мы сможем передавать в нее нужную реализацию:

```go
func writeHelloWorld(writer Writer) {  
    writer.Write([]byte("hello world"))  
}  

func main() {  
    b := &buffer{}  
    writeHelloWorld(b)  
    fmt.Println(b)  

    l := &logger{}  
    writeHelloWorld(l)  
}  

```
Таким образом, объявив функцию, которая выполняет запись `` `"Hello world"` ``, мы можем менять поведение программы, передавая разную реализацию, при этом не меняя код самой функции.

### Встраивание интерфейсов

Представьте себе ситуацию, что у вас есть два разных интерфейса (один для чтения, а другой для записи), а в какой-то момент вам потребовалось, чтобы объект обладал методами обоих интерфейсов. Можно скопировать все методы, но если мы работаем в большом проекте, такой подход может привести к ошибкам.

В Go можно использовать встраивание интерфейсов:

```go
type Reader interface {  
    Read() []byte  
}  

type Buffer interface {  
    Writer  
    Reader  
}  

func (b *buffer) Read() []byte {  
    return b.data  
}  

func useBuffer(b Buffer) {  
    writeHelloWorld(b)  
    fmt.Println(b.Read())  
}  

func main() {  
    b := &buffer{}  
    useBuffer(b)  
}  

```
### Приведение типов и работа с реализацией

Интерфейс внутри — это структура, которая хранит ссылку на исходный объект и указатели на методы интерфейса. Это позволяет легко вызывать методы и при необходимости приводить интерфейс к конкретному типу:

```go
func main() {  
    var iBuffer Buffer  

    iBuffer = &buffer{}  

    typerBuffer, ok := iBuffer.(*buffer)  
    fmt.Println(typerBuffer.data, ok)  

```
    _, ok = iBuffer.(Reader)  
    fmt.Println(ok)  
  
    _, ok = iBuffer.(interface {  
        AbsentMethod()  
    })  
    fmt.Println(ok)  
}  

### Ловушка с nil-интерфейсом

Из-за наличия ссылки внутри интерфейса можно попасть в ловушку. Что выведет следующий код?

```go
func main() {  
    var iBuffer Buffer  
    var typedBuffer *buffer  

```
    iBuffer = typedBuffer  
  
```go
    fmt.Println(reflect.TypeOf(iBuffer))  
    if iBuffer == nil {  
        fmt.Println("iBuffer =", iBuffer)  
    }

```

Кажется, что интерфейс должен быть `` `nil` ``, но это не так — потому что структура интерфейса уже содержит тип и метод-таблицу. Проверка `` `iBuffer == nil` `` вернёт `` `false` ``, даже если значение `` `nil` ``:

```go
func main() {  
    var iBuffer Buffer  
    var typerBuffer *buffer  

```
    iBuffer = typerBuffer  
      
```go
    fmt.Println(reflect.ValueOf(iBuffer).IsNil())  
    fmt.Println(reflect.ValueOf(iBuffer).MethodByName("Write").Pointer())  
}  
```

### Вызов метода на nil-значении

Даже если значение в интерфейсе `nil`, вызов метода всё равно сработает — если метод не обращается к данным объекта:

```go
func main() {  
    var iWriter Writer  
    var typerWriter *logger  

```
    iWriter = typerWriter  
  
    iWriter.Write([]byte("hello world"))  
}  

Делать так не стоит, но это работает.

Это работает из-за того, что методы на самом деле внутри - это функции, которые принимают в аргументах объект типа, на котором объявлены. В нашем случае функция `` `Write` `` помимо массива байт принимает еще и указатель на тип `` `logger` ``. Поскольку мы внутри не обращаемся к самому объекту, это работает, но стоит нам обратиться к какому- то полю объекта, то сразу же словим панику.

### Пустой интерфейс и безопасность

Поскольку интерфейс - это описание поведения типа, то что же такое пустой интерфейс? Язык позволяет нам принимать в качестве аргумента функции `` `interface{}` ``. Пустой интерфейс стоит понимать как тип `` `без ничего` ``, потому что у него нет поведения. Если же говорить про практическое применение, то долгое время пустой интерфейс был заменой типа `` `any` `` и шаблонов (которые не сразу появились в языке). Из-за того, что под пустым интерфейсом может скрываться что угодно, это порождает сложности в понимании и использовании. Из-за этого используют пустые интерфейсы в крайних случаях, когда без них совсем никак нельзя (например, сериализация).

Чтобы обезопасить себя полностью, лучше совмещать проверку на `` `nil` `` и проверку через `` `reflect` ``. При этом лучше использовать `` `IsZero` ``, а не `` `IsNil` ``, потому что в переменную могут передать объект (а не указатель) и в таком случае функция `` `IsNil` `` запаникует.

```go
if iWriter != nil && !reflect.ValueOf(iWriter).IsZero() {  
    ...  
}
```
