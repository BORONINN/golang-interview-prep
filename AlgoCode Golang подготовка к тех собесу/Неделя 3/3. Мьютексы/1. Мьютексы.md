### Проблема гонки данных (data race)

Прежде чем разбираться, что такое мьютексы и для чего они нужны, давайте посмотрим вот такой кусочек кода:

```go
func main() {  
    value := 0  
    count := 100000  

    wg := sync.WaitGroup{}  
    wg.Add(count)  

    for i := 0; i < count; i++ {  
        go func() {  
            defer wg.Done()  
            value++  
        }()  
    }  

    wg.Wait()  
    fmt.Println(value)  
}  

```
Возможный вывод
90602

При всем желании мы никогда не получим 100000, потому что тут есть состояние `` `data race` ``.

## Что такое data race

`` `Data race` `` - разновидность гонки за ресурсы, связанное с одновременным доступом к одной переменной (ресурсу) из разных горутин.

Как происходит инкремент значения на уровне ассемблера?

https://acm.bsu.by/w/images/d/d0/M7HYo.png

1. читаем значение переменной из памяти в регистр
2. увеличение значения в регистре
3. записываем значение из регистра в память

Строка `` `value++` `` включает в себя 3 операции. Как не сложно догадаться, другие потоки делают то же самое. Получается проблема: пока один поток увеличивает значение в регистре, другие могут читать старое значение из памяти.

Мьютексы позволяют избежать подобных ситуаций.

```go
func main() {  
    value := 0  
    count := 100000  

    mutex := sync.Mutex{}  

    wg := sync.WaitGroup{}  
    wg.Add(count)  

    for i := 0; i < count; i++ {  
        go func() {  
            defer wg.Done()  

```
            mutex.Lock()  
            value += 1  
            mutex.Unlock()  
        }()  
    }  
  
    wg.Wait()  
    fmt.Println(value)  
}  

Мьютекс представляет собой замок, который может закрываться и открываться, тем самым ограничивая доступ к ресурсу (в нашем случае к переменной).

После добавления мьютекса, обращение к переменной будет осуществляться последовательно, тем самым мы защищаемся от `` `data race` ``.

## Влияние мьютекса на производительность

Но защита от `` `data race` `` не бесплатная. Посмотрим, на сколько увеличится время выполнения программы после добавления мьютекса:

```go
func (p *processor) Process() {  
    c := 0  
    for range 10000 { // Имитация тупой работы  
        c++  
    }  
}  

func BenchmarkProcess(b *testing.B) {  
    p := new(processor)  
    var wg sync.WaitGroup  
    for i := 0; i < b.N; i++ {  
        for j := 0; j < 1000; j++ {  
            wg.Add(1)  
            go func() {  
                defer wg.Done()  
                p.Process()  
            }()  
        }  
        wg.Wait()  
    }  
}  

```
Результат бенчмарка

BenchmarkProcess-10        	    2112	    590725 ns/op 

Теперь добавим мьютекс и посмотрим, как изменится результат

```go
type processor struct {  
    mutex   sync.Mutex  
}  

func (p *processor) ProcessW() {  
    p.mutex.Lock()  
    defer p.mutex.Unlock()  
    c := 0  
    for range 10000 { // Имитация тупой работы  
        c++  
    }  
}  

func BenchmarkProcessW(b *testing.B) {  
    p := new(processor)  
    var wg sync.WaitGroup  
    for i := 0; i < b.N; i++ {  
        for j := 0; j < 1000; j++ {  
            wg.Add(1)  
            go func() {  
                defer wg.Done()  
                p.ProcessW()  
            }()  
        }  
        wg.Wait()  
    }  
}  

```
С мьютексом время сильно просело:

## RWMutex — компромисс между безопасностью и скоростью

В Go для подобных ситуаций был сделан `` `RWMutex` ``. Его особенность в том, что он разграничивает операции чтения и записи. Он позволяет одновременно читать из нескольких горутин, не мешая друг другу, но как только кто-то захочет записать, то только в этом случае будет выполнена полная блокировка.

```go
type processor struct {  
    rwMutex sync.RWMutex  
}  

func (p *processor) ProcessW() {  
    p.rwMutex.RLock()  
    defer p.rwMutex.RUnlock()  
    c := 0  
    for range 10000 { // Имитация тупой работы  
        c++  
    }  
}  

func BenchmarkProcessRW(b *testing.B) {  
    p := new(processor)  
    var wg sync.WaitGroup  
    for i := 0; i < b.N; i++ {  
        for j := 0; j < 1000; j++ {  
            wg.Add(1)  
            go func() {  
                defer wg.Done()  
                p.ProcessRW()  
            }()  
        }  
        wg.Wait()  
    }  
}  

```
В таком варианте работать будет ощутимо быстрее, но все равно медленнее, чем без мьютекса.

BenchmarkProcessRW-10           	    1966	    644953 ns/op

`` `RWMutex` `` стоит использовать когда много горутин читают, а одна пишет.

Использование мьютексов всегда опасно из-за дедлоков. Их легко сделать, сложно найти и невозможно забыть.

```go
func NewBox() *Box {  
    return &Box{  
        items: make(map[string]struct{}),  
    }  
}  

func (b *Box) HasItem(key string) bool {  
    b.mutex.Lock()  
    defer b.mutex.Unlock()  
    _, ok := b.items[key]  
    return ok  
}  

func (b *Box) AddItem(item string) {  
    b.mutex.Lock()  
    defer b.mutex.Unlock()  

    if b.HasItem(item) {  
        return  
    }  

    b.items[item] = struct{}{}  
}  

func main() {  
    box := NewBox()  
    box.AddItem("A")  
    box.AddItem("B")  
    fmt.Println(box.HasItem("C"))  
}  

```
## Правила безопасной работы с мьютексами

При работе с мьютексами придерживаемся следующих правил:

- вызов `` `Unlock` `` делаем через `` `defer` `` сразу после `` `Lock` `` - это поможет разблокировать мьютекс в случае паники или не потерять вызов, особенно если выходов у функции несколько
- вызывать `` `Lock` ``/`` `Unlock` `` только для манипуляции с данными. Если нужно вычислить данные и сложить их в переменную, то сперва вычисляем, а потом под локом меняем значение
- если есть потребность в размещении некоторой логики под локом, то убеждаемся, что вызываемые функции внутри не выполняют новые локи (как пример с коробкой)
- если можно обойтись без мьютекса - лучше обойтись без мьютекса (например использовать каналы или атомики)
