## Как работают каналы

Каналы — важный функционал в языке Go, используемый для коммуникации между различными горутинами. Каналы - одна из тех фичей, за которые все так любят Go. Использование каналов позволяет избежать сложного совместного использования памяти (как например при использовании мьютексов), делая проектирование параллельных программ проще и безопаснее.

Основное предназначение каналов - передать данные из одной части программы в другую. Но это не единственное применение (поговорим дальше).

Использование канала выглядит следующим образом:

```go
func main() {  
    ch := make(chan int, 2)  

```
    ch <- 1  
    ch <- 2  
  
```go
    fmt.Println(<-ch)  
    fmt.Println(<-ch)  
}  
```
Создаем канал с нужным нам типом, передаем данные внутрь, затем мы их можем прочитать. Каналы работают по схеме FIFO.

Внимательные могли заметить, что при создании канала был указан размер.

Дело в том, что каналы бывают буферизированные, а бывают небуферизированные. Разница в поведении при записи/чтении. Рассмотрим оба варианта подробнее.

### Буферизированные каналы

```go
func main() {  
    ch := make(chan int, 2)  

```
    ch <- 1  
    ch <- 2  
    ch <- 3 // В данный момент выполнение будет приостановлено  
}  

Когда количество данных внутри достигло размера канала, попытка записать новое значение в канал приведет к блокировке горутины до тех пор, пока из канала не будет прочитано хотя бы одно значение.

Аналогичная ситуация и с чтением:

```go
func main() {  
    ch := make(chan int, 2)  

```
    ch <- 1  
    ch <- 2  
  
```go
    fmt.Println(<-ch)  
    fmt.Println(<-ch)  
    fmt.Println(<-ch) // В данный момент выполнение будет приостановлено  
}  
```
Таким образом, чтение из канала будет осуществляться до тех пор, пока в нем есть значения, а запись, пока там есть место.

### Небуферизированные каналы

Небуферизированный канал - это канал с размером ноль. Т.е. при чтении и записи в канал происходит блокировка выполнения горутины.

```go
func main() {  
    ch := make(chan int) // Небуферизированный канал создается без указания длины  

```
    ch <- 1 // В данный момент выполнение будет приостановлено  
}  

## Синхронизация с использованием каналов

А как же тогда ими пользоваться? Просто и удобно! Каналы создавались для передачи данных между горутинами.

Если мы хотим прочитать или записать в небуферизированный канал, то мы блокируемся до момента, пока с другой стороны канала не появится кто-то, кто захочет прочитать данные из него.

```go
func main() {  
    startAt := time.Now()  
    ch := make(chan time.Duration)  

    go func() {  
        fmt.Println("start func")  
        ch <- time.Since(startAt)  
        fmt.Println("meddle in func")  
        ch <- time.Since(startAt)  
        fmt.Println("finish func")  
    }()  

    fmt.Println("1 since ", <-ch)  
    fmt.Println("start wait")  
    time.Sleep(5 * time.Second)  
    fmt.Println("finish wait")  
    fmt.Println("2 since ", <-ch)  
    fmt.Println("finish at ", time.Since(startAt))  
}  

```

**Вывод** **будет следующим**

start func
meddle in func
1 since 50.709µs
start wait
finish wait
finish func
2 since 53µs
finish at  5.001104292s

Можно заметить, что вывод `` `finish func` `` выводится уже после `` `finish wait` ``, но время выведенное в `` `2 since` `` всего лишь несколько микросекунд.

Это происходит из-за того, что сперва вычисляется `` `time.Since(startAt)` ``, а потом значение мы пытаемся записать в канал.

При записи мы блокируемся, пока выполнение основной функции не дойдет до чтения.

Итого у нас получается 2 точки синхронизации программы:

1. первая - запись в канал и чтение fmt.Println("1 since ", <-ch)
2. вторая - запись в канал и чтение fmt.Println("2 since ", <-ch)

Таким образом, каналы можно использовать еще и для синхронизации хода выполнения каких то действий.

Например:

Что выведет данный код? 30 или 40?

```go
func main() {  
    values := 10  

    go func() {  
        values += 10  
    }()  

    go func() {  
        values *= 2  
    }()  

    time.Sleep(1 * time.Second)  
    fmt.Println(values)  
}  

```

Правильного ответа тут нет. Все зависит от порядка, в котором были выполнены операции. Эта ситуация называется race condition.

Race condition - это ошибка, при которой время или порядок событий влияет на корректность части кода. В большинстве случаев, race condition - это следствие ошибки проектирования.

Добавим сюда каналы, чтобы получить 40

```go
func main() {  
    values := 10  

    ch1 := make(chan struct{})  
    ch2 := make(chan struct{})  

    go func() {  
        values += 10  
        ch1 <- struct{}{}  
    }()  

    go func() {  
        <-ch1  
        values *= 2  
        ch2 <- struct{}{}  
    }()  

```
    <-ch2  
    fmt.Println(values)  
}  

И для того, чтобы получить 30

```go
func main() {  
    values := 10  

    ch1 := make(chan struct{})  
    ch2 := make(chan struct{})  

    go func() {  
        <-ch1  
        values += 10  
        ch2 <- struct{}{}  
    }()  

    go func() {  
        values *= 2  
        ch1 <- struct{}{}  
    }()  

    <-ch2  
    fmt.Println(values)  
    
} 
```


Как видно, теперь порядок строк детерминирован и мы можем не использовать sleep, чтобы дождаться горутины.

Таким же образом для синхронизации каналы используются в реализации тикера (time.Ticker) и контексте с отменой

## Тикер

```go
func main() {  
    ticker := time.NewTicker(time.Second)  
    start := time.Now()  

    for {  
        <-ticker.C // С - небуферизированный канал  
        fmt.Println(time.Since(start))  
    }  
}  

```
1.001051458s
2.001037458s
3.001065708s
4.001019166s
5.000375625s

## Контекст

```go
func main() {  
    ctx, cancel := context.WithTimeout(context.Background(), time.Second)  
    defer cancel()  
    started := time.Now()  

    fmt.Println(time.Since(started))  
    <-ctx.Done()  
    fmt.Println(time.Since(started))  
}  

```
42ns
1.001043917s

Но давайте вернемся к передаче данных через каналы.

Каналы поддерживают перебор значений в канале через цикл. При этом цикл будет прерван автоматически, когда канал будет закрыт и будут перебраны все значения

```go
func main() {  
    ch := make(chan int)  

    go func() {  
        for i := 0; i < 10; i++ {  
            ch <- i  
        }  

```
        close(ch)  
    }()  
  
    for v := range ch {  
        fmt.Println(v)  
    }  
}  

## Основные правила работы с каналами

Стоит запомнить несколько основных правил работы с каналами:

||Чтение|Запись|
|---|---|---|
|Открытый канал (пустой)|Блокировка|Запись без блокировки|
|Открытый канал (полный)|Чтение без блокировки|Блокировка|
|Закрытый канал|Вернется нулевое значение указанного типа|Паника|
|Неинициализированный канал|Блокировка|Блокировка|

Чтобы проверить, закрыт канал или нет, вместе со значением из канала возвращается флаг, было это значение реальным, или канал уже закрыт и мы получаем нулевые значения:

```go
func main() {  
    ch := make(chan int, 1)  

```
    ch <- 1  
    val, ok := <-ch  
    fmt.Println(val, ok)  
  
    ch <- 2  
    close(ch)  
  
    val, ok = <-ch  
    fmt.Println(val, ok)  
  
    val, ok = <-ch  
    fmt.Println(val, ok)  
}  

1 true
2 true
0 false

Учитывая это, и то, что нельзя писать в закрытый канал, есть такое правило, что канал должен закрывать producer (продюсер) данных. Это не является ультиматумом, все отдается на откуп разработчику, но такой подход позволяет избежать многих часов отладки кода.

## Select — множественный выбор по каналам

Дальше рассмотрим еще один важный инструмент при работе с каналами - это конструкция `` `select` ``.

По своей сути она схожа со структурой `` `switch` ``, но предназначена для работы с каналами.

```go
func doSomething(ch chan<- int) {  
    ...  
}  

func main() {  
    ch := make(chan int, 1)  

```
    doSomething(ch)  
  
    <-ch  
}  

У нас есть некоторая функция, куда мы передаем наш канал, но на выходе мы не знаем, было туда что-то записано или нет.

Как следствие, при чтении мы можем получить значение, а можем заблокироваться (ведь в канале ничего нет).

Для таких случаев и нужен `` `select` ``

```go
func doSomething(ch chan<- int) {  
    //ch <- 1  
}  

func main() {  
    ch := make(chan int, 1)  

```
    doSomething(ch)  
  
    select {  
    case i := <-ch:  
        fmt.Println("received", i)  
    default:  
        fmt.Println("nothing")  
    }  
}  

Если значение в канале было и мы можем его прочитать, то будет вызвана ветка `` `case` ``. Иначе мы свалимся в `` `default` `` и чтения из канала не будет.

В этом примере при передаче канала в функцию было явно указано, что в такой канал можно только писать. Синтаксис языка позволяет в аргументах функций указать, как будет использоваться канал:

- только на чтение (`` `<- chan int` ``)
- только на запись (`` `chan <- int` ``)
- без ограничений (`` `chan int` ``)

`` `Select` `` можно использовать с несколькими каналами.

Например это часто используется, когда нужно по тикеру выполнять какую-то операцию, но при этом поддерживать прерывание по таймауту:

```go
func main() {  
    ticker := time.NewTicker(time.Second)  

    ctx, cancel := context.WithTimeout(context.Background(), time.Second*5)  
    defer cancel()  

```
loop:  
    for {  
        select {  
        case <-ticker.C:  
            fmt.Println("tick")  
        case <-ctx.Done():  
            fmt.Println("timeout")  
            break loop  
        }  
    }  
  
```go
    fmt.Println("finish")  
}  
```
tick
tick
tick
tick
tick
timeout
finish

Тут использован лейбл, потому что break так же используется и для `` `select` `` (и в данном случае без лейбла он относился бы к `` `select` ``)

## Deadlock — когда всё встало

Ситуация, когда мы пытаемся прочитать из канала, в который никто не пишет, называется `` `deadlock` `` .

Deadlock - это ситуация, когда одна или несколько горутин ожидают ресурс, доступ к которому уже не будет предоставлен. Обычно в качестве примера приводят два потока с двумя мьютексами, но получить deadlock можно и куда более простым способом:

```go
func main() {  
    ch := make(chan int)  
    <-ch  
}  

```
Если запустить такой код, то в консоли можно будет увидеть те сами заветные слова

fatal error: all goroutines are asleep - deadlock!

Но не всегда так легко обнаружить дедлок. Данный пример синтетический, а в реальных проектах дедлок обычно проявляется в виде увеличения latency, роста потребления памяти/cpu (утечка горутин), роста ошибок.

Как и `` `race condition` ``, `` `deadlock` `` - это ошибка проектирования. Получить `` `deadlock` `` можно на любом средстве синхронизации, поэтому стоит быть начеку всегда.
