## Введение

`WorkerPool` — шаблон для достижения параллельности в Go. Идея заключается в запуске фиксированного количества горутин, которые ожидают назначения работы. Когда задача поступает работнику, она выполняется в фоновом режиме, позволяя основной горутине продолжать выполнение другого кода.

Примером этого паттерна из жизни можно считать McDonald's: кассир принимает заказы, которые попадают в электронную очередь к поварам. Свободный повар начинает готовить бургеры из очередного заказа. После приготовления бургеры попадают на ленту, откуда их забирает упаковщик, который отдает собранный заказ.

## Структура WorkerPool

## Job

Полезная нагрузка, которую должен выполнить воркер. Это может быть вычисление, запрос в сеть или любая другая работа, которую нужно выполнить независимо от остального.

```go
func (j Job) execute(ctx context.Context) Result {
    value, err := j.ExecutionFn(ctx)
    if err != nil {
        return Result{
            Err: err,
        }
    }

    return Result{
        Value: value,
    }
}

```
## Очередь задач

Канал, который используется для хранения входящих задач. Через него поступают новые задачи воркерам.

```go
func (wp WorkerPool) Add(jobsBulk []Job) {
    for i := range jobsBulk {
        wp.jobs <- jobsBulk[i]
    }
    close(wp.jobs)
}

```
## Worker

Работник, который выполняет задачи. Его основная и единственная задача — выполнить переданную работу и отправить результат.

```go
func worker(ctx context.Context, wg *sync.WaitGroup, jobs <-chan Job, results chan<- Result) {
    defer wg.Done()
    for {
        select {
        case job, ok := <-jobs:
            if !ok {
                return
            }
            // Выполняем джобу и отправляем результат
            results <- job.execute(ctx)
        case <-ctx.Done():
            fmt.Printf("cancelled worker: %v\n", ctx.Err())
            return
        }
    }
}

```
## Менеджер воркеров

Управляет запуском воркеров и корректным завершением их работы.

```go
func (wp WorkerPool) Run(ctx context.Context) {
    var wg sync.WaitGroup
    wg.Add(wp.workersCount)

    for i := 0; i < wp.workersCount; i++ {
        go worker(ctx, &wg, wp.jobs, wp.results)
    }

    wg.Wait()
    close(wp.results)
}

```
## Пример использования

Для демонстрации создадим джобы, имитирующие вычисления:

```go
func makeJob(value int) workerpool.Job {
    return workerpool.Job{
        ExecutionFn: func(_ context.Context) (int, error) {
            time.Sleep(time.Duration(value) * time.Second)
            return value, nil
        },
    }
}

```
Добавим как успешные джобы, так и одну с ошибкой для демонстрации:

Go

var jobs []workerpool.Job

for v := 0; v < 10; v++ {
    jobs = append(jobs, makeJob(v))
}

jobs = append(jobs, workerpool.Job{
    ExecutionFn: func(_ context.Context) (int, error) {
        return 0, errors.New("job failed")
    },
})

Запустим пул и проверим время обработки всех джоб:

```go
wp := workerpool.New(4)

ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()

start := time.Now()

```
go wp.Add(jobs)
go wp.Run(ctx)

for res := range wp.Results() {
    fmt.Println(res)
}

```go
fmt.Println(time.Since(start))
```

## Как это работает

Когда мы передаем массив джобов в воркерпул, они попадают в очередь, из которой их забирают воркеры. Воркеры выполняют джобы и отправляют результаты в канал `` `results` ``.

Если джобов больше, чем размер пула (что весьма вероятно), джобы будут постепенно поступать в канал, а воркеры будут обрабатывать их со своей скоростью.

Как только в очередь `` `jobs` `` будут записаны все входные данные, канал закрывается. Это позволяет воркерам понять, что обрабатывать больше нечего, и они завершаются штатным способом.

После завершения всех воркеров менеджер дожидается завершения `WaitGroup` и закрывает канал `` `results` ``. Таким образом мы сообщаем потребителю, что все вычисления завершены.

Методы `` `Add` `` и `` `Run` `` необходимо запускать в отдельных горутинах от считывания результатов, потому что при ограниченном размере пула мы можем попасть в `deadlock`.
