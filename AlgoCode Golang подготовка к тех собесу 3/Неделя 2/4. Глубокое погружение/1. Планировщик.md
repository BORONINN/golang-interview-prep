## Введение

Вот мы и добрались до изюминки go - это горутины.

Горутины дают возможность запускать функции конкурентно. Так же горутины называют green thread или легковесные потоки. Но потоками они не являются.

Горутины работают поверх системных потоков. Т.е. горутины являются некоторой абстракцией того, что должно выполняться в рамках системного потока в данный момент. А выбирает, что будет выполняться, шедулер (scheduler) Go.

Ничего не напоминает? Правильно - это похоже на системные треды и планировщих ОС.

```go
func work(n int) {
    fmt.Println("start work", n)
    time.Sleep(time.Second*10)
    fmt.Println("end work", n)
}

func main() {
    for n := range 5 {
        work(n)
    }
}

```
Возьмем вот такой код и очевидно, что мы получим вывод:

start work 0
end work 0
start work 1
end work 1

Добавим сюда запуск функции в отдельной горутине:

```go
for n := range 5 {
    go work(n)
}

```
и вывод уже будет интереснее

start work 0
start work 4
start work 3
start work 2
start work 1
end work 0
end work 1
end work 4
end work 3
end work 2

В целом тут ничего удивительного пока нет, мы запустили код параллельно и он выполнился параллельно. При этом время выполнения будет составлять ~10 секунд.

## Планировщик Go: G-M-P модель

Давайте теперь копнем чуть глубже, чтобы разобраться в том, как же именно горутины соотносятся с системными потоками.

Планировщик Go оперирует тремя сущностями: горутинами (G), машинами (M) и процессорами (P)

G - горутины, которые мы можем запускать в коде. Действует как наименьшая исполнительная единица. Сама горутина - это тоже структура, которая инициализируется в момент вызова ключевого слова `go func()` . А при завершении, горутины не удаляются, а помещаются в пул.

Горутина может находиться в нескольких состояниях:

- Ожидание - горутина ждет, пока выполнится некоторая операция (синхронизация на мьютексах, сискол и тп)
- Готова к выполнению - горутина находится в очереди на выполнение
- Запущена - горутина выполняется в потоке M.

P - процессор, но не физический, а логический. Это абстракция для управления ресурсами. По умолчанию количество процессоров равно количеству ядер процессора, но их можно изменять. Именно это мы делали в последнем примере через функцию `runtime.GOMAXPROCS`.

Каждый процессор имеет собственный список запускаемых горутин. Это называется локальная очередь запуска. Именно сюда попадают горутины, когда мы их запускаем. При этом размер локальной очереди ограничен 256 горутинами. Как только она переполняется, появляется общая очередь (глобальная очередь).

M - машина или поток операционной системы. Представляет собой ресурсы ОС.

## Ограничение системных потоков

Когда горутина готова к работе, ей выделяется системный поток. Если уже есть существующий, будет назначен существующий, но если все потоки уже заняты/заблокированы (системными вызовами и т.п.), то в этом случае планировщик создаст новый поток для этой горутины. Но стоит учитывать тот факт, что программа на Go может создать не более 10к системных потоков:

## Взаимодействие MPG

Давайте теперь разберем подробнее взаимодействие MPG

1. Инициализация горутины. Когда мы хотим запустить функцию в горутине `go func()` , среда выполнения либо создает новую горутину, либо выбирает существующую из пула.
2. Назначение очереди. Горутина ищет свое место в очереди, и если локальные очереди всех логических процессоров (P) заполнены, то она помещается в глобальную очередь.
3. Сопряжение потоков. Машина (M) захватывает P и начинает обработку горутины из локальной очереди. Поскольку M взаимодействует с горутиной, то связанный с ней P становится недоступным для других M.
4. Кража. Когда у P заканчиваются G для исполнения, он случайным образом выбирает другой P и пытается украсть у него половину горутин из его очереди. Если это сделать не получилось, он проверяет глобальную очередь, а после этого следует netpoller.
5. Выделение ресурсов. После того, как M выбирает горутину, он обеспечивает все необходимые ресурсы для запуска G.

Если во время работы G системный поток блокируется (например из-за системного вызова), M будет ожидать выполнения. Мы получаем ситуацию, когда часть ресурсов простаивает. В этом случае горутина помечается как заблокированная, планировщик отсоединяет остановленную M от ее P и подключает другую, готовую горутину из очереди к новой или существующей M, которая затем закрепляется за P. Этот процесс (отключение горутины) называется парковка (parking) горутины.

## Прерывания

В Go горутины работают в режиме **кооперативной многозадачности**, но с версии **1.14** была добавлена поддержка **вытесняющей (preemptive) многозадачности**, чтобы избежать ситуаций, когда одна горутина надолго занимает поток (M), не давая выполняться другим.

**1. Кооперативная многозадачность (до Go 1.14)**

Раньше горутина могла выполняться, пока явно не отдавала управление, например:

- При вызове `` `runtime.Gosched()` `` (явное переключение).
- При блокирующих операциях (каналы, мьютексы, системные вызовы).

**Проблема:**

Если горутина выполняла долгий цикл без вызовов блокирующих операций, другие горутины могли **голодать** (не получать CPU).

**2. Вытесняющая многозадачность (с Go 1.14)**

Теперь планировщик **принудительно прерывает** долгие горутины, даже если они не блокируются.

Как это работает?

- В фоне работает **sysmon** (системный монитор), который следит за горутинами.
- Если горутина выполняется **дольше 10 мс**, sysmon помечает её как **подлежащую вытеснению**.
- Планировщик вставляет **асинхронный прерывающий сигнал** (например, `` `SIGURG` `` в Unix).
- При обработке сигнала горутина **приостанавливается** и возвращается в очередь, давая шанс другим.

```go
func main() {
    runtime.GOMAXPROCS(1) // Один P для наглядности

    go func() {
        for i := 0; ; i++ {} // Бесконечный цикл
    }()

    time.Sleep(time.Millisecond)
    fmt.Println("Другая горутина получила CPU!")
}

```
До Go 1.14 программа зависла бы в цикле. Сейчас sysmon прервёт цикл, и fmt.Println выполнится.

## Балансировка между локальной и глобальной очередями

По умолчанию планировщик отдает приоритет локальным очередям процессоров (P) — это обеспечивает эффективное использование кэша процессора и уменьшает накладные расходы на синхронизацию. Однако такой подход может привести к дисбалансу: пока одни процессоры загружены, горутины в глобальной очереди могут долго оставаться без выполнения.

Чтобы предотвратить эту проблему, планировщик использует **периодическую проверку глобальной очереди**: каждые 61 такт работы потока (M) он принудительно заглядывает в глобальную очередь и забирает оттуда задачи, если они есть.

**Почему именно 61?**

- Это **простое число**, что помогает избежать резонансных эффектов при работе нескольких потоков.
- **Компромисс между частотой и накладными расходами:**
    - Слишком частая проверка (например, каждые 10 тактов) увеличивала бы нагрузку на синхронизацию.
    - Слишком редкая (например, каждые 100+) могла бы привести к заметным задержкам в распределении задач.

Таким образом, значение 61 выбрано эмпирически как оптимальное для баланса между эффективностью и справедливостью.
