## Проблема без использования WaitGroup

`` `WaitGroup` `` - это механизм синхронизации, который блокирует выполнение программы до завершения горутин из определённой группы.

```go
func main() {
    for i := range 5 {
        go func() {
            time.Sleep(time.Second * 2)
            fmt.Printf("finish goroutine %d\n", i)
        }()
    }

    fmt.Println("finish main")
}

```
Вот такой простой пример проблемы, которую решает `` `waitGroup` ``. В выводе программы мы не увидим `` `finish goroutine` `` из-за того, что программа уже завершит свое выполнение к тому моменту, когда выполнение может дойти до вывода этой строки.

Но если мы добавим `` `WaitGroup` ``, то все встанет на свои места:

```go
func main() {
    wg := sync.WaitGroup{}

    for i := range 5 {
        wg.Add(1)
        go func() {
            defer wg.Done()

            time.Sleep(time.Second * 2)
            fmt.Printf("finish goroutine %d\n", i)
        }()
    }

    wg.Wait()

    fmt.Println("finish main")
}

```
## Что происходит при использовании WaitGroup

По выводу мы можем убедиться, что сперва мы дождались выполнения горутины и только после этого напечатали `` `finish main` ``

Сам `` `waitGroup` `` внутри устроен следующим образом:

1. Функцией `` `Add` `` мы задаем количество горутин, которые мы хотим дождаться (увеличиваем счетчик внутри `` `waitGroup` ``)
2. При вызове `` `Done` `` мы уменьшаем счетчик
3. Функция `` `Wait` `` дожидается, пока счетчик дойдет до нуля и только после этого возвращает выполнение обратно

## Рекомендации и подводные камни

Вызов `` `Done` `` лучше делать через `` `defer` `` (так же для мьютексов, каналов и т.п.) на случай, чтобы не потерять его в конце лямбды или на случай паники.

Стоит помнить, что если количество вызовов `` `Add` `` и `` `Done` `` не будет совпадать, то можно поймать панику (если будет лишний `` `Done` ``) или `` `deadlock` `` (если был лишний `` `Add` ``).
