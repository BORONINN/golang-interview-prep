## Введение в сетевое программирование на Go

Работа с сетью в Go удобная. Это и не мудрено, ведь именно на это был сделан упор при его разработке.

Давайте посмотрим как будет выглядеть микро сервер с клиентом

```go
func serve(ln net.Listener) {
    for {
        conn, err := ln.Accept()
        if err != nil {
            continue
        }

        data := make([]byte, 1024)
        dataLen, err := conn.Read(data)
        if err != nil {
            fmt.Printf("failed to read: %v\n", err)
            os.Exit(1)
        }

        fmt.Print("read data - ", string(data[:dataLen]))
        return
    }
}

func client() {
    conn, err := net.Dial("tcp", ":8081")
    if err != nil {
        fmt.Printf("failed to create connection: %v\n", err)
        os.Exit(1)
    }
    defer conn.Close()

    writeLen, err := conn.Write([]byte("hello world"))
    if err != nil {
        fmt.Printf("failed to write: %v\n", err)
        os.Exit(1)
    }

    fmt.Printf("wrote %d bytes\n", writeLen)
}

func main() {
    ln, err := net.Listen("tcp", ":8081")
    if err != nil {
        fmt.Println(err)
        return
    }
```

## Блокирующее чтение и магия под капотом

Мы поднимаем `` `tcp` `` сервер и начинаем слушать порт. В отдельном потоке мы подключаемся к нашему же серверу и отправляем туда данные. Кажется, что код чтения/записи синхронный и блокирующий. Но это не совсем так.

Когда мы пытаемся выполнить операцию чтения (например), мы не знаем, пишет кто-то с той стороны, если ли уже данные в буфере или нужно подождать, от нас все это скрыто. Стучаться в сокет и постоянно спрашивать “Есть че? А сейчас? А теперь?” будет, мягко говоря, не эффективно. Нам было бы удобнее, чтобы нам кто-то сообщил о том, что данные есть и их можно прочитать, а до этого момента нам особо делать нечего.

## Как работает netpoller в Go

Примерно так работает netpoller в Go. Когда мы пытаемся прочитать данные из файлового дескриптора (сокета), при наличии этих данных будет выполнено чтение, но если их нет (или мы вычитали все что было), горутина блокируется до поступления данных. А этот момент отслеживается на уровне ОС. В unix это `` `epoll` ``, в windows - `` `IoCompletionPort` ``. После блокировки горутины `netpoller` запоминает, какая горутина ждет данный файловый дескриптор (ФД). Когда `netpoller` получает уведомление от ОС о том, что можно выполнить операцию ввода/вывода над ФД, то он проверяет, есть ли горутины, заблокированные на этом ФД и уведомляет их, если такие есть. После этого, горутина сможет повторить операцию.

## Зачем всё так устроено

Можно задаться вопросом “а зачем так сложно?”.

Тут есть много причин:

- во-первых, код, который получается в итоге, более понятный (обилие коллбеков / промисов зачастую приводит к тому, что в коде сложно разобраться и найти концы)
- во-вторых, мы имеем один поток, который отслеживает возможность выполнять операции ввода/вывода и сигнализирует об этом. А реальная работа происходит в отдельных горутинах.
- в-третьих, использование `` `epoll` `` (и подобных) уменьшает количество системных вызовов, что увеличивает производительность
