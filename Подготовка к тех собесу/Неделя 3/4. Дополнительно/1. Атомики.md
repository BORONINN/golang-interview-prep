## Пример проблемы без атомиков

Прежде чем разбираться с атомиками, посмотрим на такой код

```go

var (
    flag  = false
    value = 10
)

func changeFunc() {
    value = 15
    flag = true
}

func main() {
    go changeFunc()

    for !flag {
    }

    println(value)
}
```
Вроде бы все нормально. Но так ли это?

## В чем проблема

На первый взгляд, может показаться, что проблем с таким кодом, потому что пишем мы только в одной горутине, а читаем в другой.

Но палки в колеса нам вставляют оптимизации компилятора. Компилятор может преобразовывать код и переставлять местами инструкции. Т.е. мы не можем быть уверены, что в горутине мы сперва поменяем значение переменной `` `value` `` и только потом возведем флаг. Потому что на деле эта последовательность может быть изменена и мы даже не будем об этом знать. Компилятор не со всеми инструкциями так может делать. Например, если одна строка зависит от другой, то они сохранят последовательность. Но если зависимости нет (в области видимости), то перестановка возможна.

Если мы запустим race detector, то он покажет 2 гонки:

1. взаимодействие с переменной `` `value` ``
2. взаимодействие с переменной `` `flag` ``

## Что такое барьеры памяти

Помимо “помощи” от компилятора, нам могут усложнить ситуацию еще и особенности работы процессора.

В этот момент к нам на помощь приходят барьеры памяти. Если говорить простым языком, то барьер памяти - это невидимая линия, через которую запрещено перемещать инструкции. Это не означает, что инструкции не будут переставляться, но это гарантирует, что инструкции останутся с той стороны барьера, где они были изначально.

В нашем примере было бы логично поставить барьер между присвоением переменным значений

```go
func changeFunc() {
    value = 15
    // ---------- барьер
    flag = true
}

```
Барьеры могут действовать только на операции чтения, только на операции записи и на все операции. Но поскольку Go - простой язык, нам доступен только full барьер (чтение и запись). Это значит, что все операции чтения и записи, которые были до барьера, будут выполнены до первой операции, после барьера.

Барьеры памяти не отменяют оптимизаций компилятора/процессора. Они гарантируют последовательность инструкций относительно барьера.

## Как работают атомики

Атомики в Go под капотом используют барьеры памяти, чтобы обеспечить атомарность операций над памятью. Внутреннее устройство атомиков наружу не торчит, но есть сноска на официальном сайте: “все атомарные операции в программе ведут себя так, как будто выполняются в некотором согласованном порядке”.

Это значит, что мы можем использовать атомики в разных горутинах и не бояться, что мы поймаем `` `data race` ``.

```go

var (
    flag  = atomic.Bool{}
    value = 10
)

func changeFunc() {
    value = 15
    flag.Store(true)
}

func main() {
    go changeFunc()

    for !flag.Load() {
    }

    println(value)
}
```
Мы можем переписать предыдущий пример с использованием атомиков и data race в нем не будет, потому что порядок инструкций гарантирован барьером в атомике.

## Когда использовать атомики

На практике атомики используют нечасто, потому что это низкоуровневый инструмент. Но есть ряд задач, где атомики позволяют написать безопасный код и более быстрый, чем код, реализованный через мьютексы.
